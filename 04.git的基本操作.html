1.获取git仓库的两种方式 1.将尚未进行版本控制的本地目录转换为Git仓库 1.在项目目录中,通过鼠标右键打开"Git Bash" 2.执行git init命令将当前的目录转化为Git仓库 git init命令会创建一个名为.git的隐藏目录,这个.git目录就是当前的Git仓库 2.从其他服务器克隆一个已存在的Git仓库 工作区中文件的4种状态 丰富 工作区中的每一个文件可能有4种状态,这四种状态共分为两大类： (未跟踪untracked)不被Git所管理的文件 <=> 未被Git管理 <===> 已被Git管理 => 3个状态 {未修改(unmodified) 工作区中的文件的内容和Git仓库中文件的内容保持一致, 已修改(modified)工作区中的文件的内容和Git仓库中文件的内容不一致, 已暂存(staged)工作区中的文件已被放到暂存区,准备将修改后的文件保存到Git仓库中} Git的最终操作结果:让工作区中的文件处于"未修改"状态 2.git status 检查文件处于什么状态 精简的方式 -s 是 --short的简写; 精简的方式显示文件的状态 git status -s git status --short 未跟踪的文件前面有红色的??标记 3.跟踪新文件 git add命令跟踪新文件 //git add index.html index.html文件在changes to committed这行的下面,说明已经被跟踪并处于暂存状态 精简的方式显示文件的状态: 新添加到暂存区中的文件前面有绿色的A标记 4.提交更新 现在暂存区中有一个index.html文件等待提交到git仓库中保存,可以执行git commit命令进行提交, 其中-m选项后面是本次的提交消息,用来对提交的消息做进一步的描述: git commit -m "新建了index.html文件" 5.对已提交的文件进行修改 目前index.html文件已被Git跟踪,并且工作区和Git仓库中的index.html文件保持一致,当我们修改了工作区中的index.html中的内容后, 再次运行git status/git status -s命令,显示:文件index.html出现在Changes not staged for commit这行下面,说明已跟踪内容发生、 变化,还没有放到暂存区 修改过的、没有放到暂存区的文件前面有红色的M标记 6.暂存已修改的文件 目前工作区中的文件已被修改,如果要暂存这次修改,需要再次运行git add命令 git add命令有3个功能: 1.开始跟踪新文件 2.把已跟踪的,且已修改的文件放到暂存区 3.把有冲突的文件标记为已解决状态 7.提交已暂存的文件 再次运行git commit -m "提交信息" 8.撤销对文件的修改 撤销对文件的修改指的是: 把对工作区中对应文件的修改,还原成Git仓库中所保存的版本。 操作的结果:所有的修改会丢失,且无法恢复！危险高 git checkout -- index.html 本质是Git仓库中的文件内容覆盖工作区中已修改的内容 9.向暂存区中一次性添加多个文件 使用命令git add . 10.取消暂存的文件 使用git reset HEAD 要移除的文件名称 /git reset HEAD . 多文件移除 11.跳过使用暂存区 命令git commit -a -m "描述消息" Git就会自动把所有已经跟踪的文件暂存起来一并提交,从而跳过git add步骤 12.移除文件 从Git仓库中移除文件的方式有2种: 1.命令git rm -f "文件名" => 从工作区和Git仓库中同时移除对应的文件 2.命令git rm --cached "文件名" => 只从Git仓库中移除文件 13.忽略文件 一般有些文件无需纳入Git的管理,也不希望他们总出现在未跟踪文件列表中,此情况下,我们可以创建一个名为.gitignore的配置文件,列出要忽略的文件的匹配模式。 文件.gitignore的格式规范如下: 1.以#开头的是注释 2.以/结尾的是目录 3.以/开头防止递归 4.以！开头表示取反 5.可以使用glob模式进行文件和文件夹的匹配(glob指简化了的正则表达式) 1. *.a => 代表忽略所有以.a结尾的文件 2. !lib.a => 代表跟踪所有的lib.a文件 3. /TODO => 代表忽略当前目录下的TODO文件 而不忽略subdir/TODO(其他目录下的TODO) 4. build/ => 忽略所有目录下名为build的文件夹 //忽略build目录下的所有文件 5. doc/*.txt => 忽略doc/notes.txt文件 但不忽略 doc/serve/notes.txt文件 6. doc/**/*.txt => 忽略doc目录下子目录下所有的.txt文件 14.查看提交历史 使用git log命令 键盘q退出 按时间先后顺序列出所有的提交历史,最近的提交排在最上面 git log 只展示最新的两条提交历史,数字可以按需进行填写 git log -2 在一行上展示最近两条提交历史的信息 git log -2 --pretty=oneline 在一行上展示最近两条提交历史的信息,并自定义输出的格式 %h 提交简写的哈希值 %an作者名字 %ar作者修订日期 按多久以前的方式显示 %s提交说明 git log -2 --pretty=format:"%h | %an | %ar | %s" 15.回退到指定的版本 在一行上展示所有的提交历史git log --pretty=oneline 使用git reset --hard命令,根据指定的提交 ID 回退到指定的版本 git reset --hard 在旧版本中使用git reflog --pretty=oneline命令,查看命令操作的历史 git reflog --pretty=oneline 再次根据最新的提交ID,跳转到最新的版本 git reset --hard
